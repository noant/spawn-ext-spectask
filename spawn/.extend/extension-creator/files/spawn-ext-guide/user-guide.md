# Spawn extension author guide

## 1. Concepts

**Extension** — an installable pack of methodology: template files, agent skills, optional MCP servers, optional Python setup hooks, ignore rules.

**Extension repository** — any git repo (or folder/zip) that contains an **`extsrc/`** directory.

**Target repository** — the project where you run `spawn init` and `spawn extension add ...`. After install, Spawn writes under that repo’s root, e.g. `spawn/.extend/<name>/`.

As an author you separate:

- **Static** content: safe for the tool to overwrite on extension update (shared methodology).
- **Artifact** content: owned by the project; must not be overwritten on update.
- **Rendered outputs**: navigation index, IDE skills, IDE MCP config fragments — generated by Spawn from your declarations.

---

## 2. Directory layout

Only **`extsrc/`** is copied into the target as the installed extension. The rest of your extension repo is for your own development.

```text
your-extension-repo/
  extsrc/
    config.yaml       # required
    mcp/              # optional; cross-platform MCP (see §6)
      windows.json
      linux.json
      macos.json
    skills/
      *.md            # skill sources; filename = key in config.yaml
    files/
      ...             # mirrors paths declared under config files:
    setup/
      *.py            # optional lifecycle scripts
```

**Rules:**

- **`extsrc/config.yaml`** is required.
- Every file under **`extsrc/files/`** must appear as a key under **`files:`** in `config.yaml`. With **`spawn extension check . --strict`**, undeclared files under `extsrc/files/` are errors.

---

## 3. Extension id and install location **`config.yaml` → `name`**

Field **`name`**: stable identifier, prefer **`kebab-case`**, lowercase.

Installed layout:

```text
<target-root>/spawn/.extend/<name>/
```

If **`name`** is omitted, install may fall back to the placeholder name **`extension`** — always set **`name` explicitly** and keep it stable across releases.

---

## 4. `config.yaml` — full schema

Use **`schema: 1`** for the current format.

### 4.1 Top-level keys

| Key | Required | Type | Purpose |
|-----|----------|------|---------|
| `schema` | yes | int | Must be **`1`**. |
| `version` | yes | string | Extension version (string comparison on update). |
| `name` | strongly recommended | string | Id and folder name `spawn/.extend/{name}/`. |
| `files` | no | map | Target-relative paths → file metadata. |
| `folders` | no | map | Target-relative directory paths → folder metadata. |
| `skills` | no | map | Skill filename under `skills/` → skill metadata. |
| `agent-ignore` | no | list of strings | Glob patterns merged into IDE agent-ignore. |
| `git-ignore` | no | list of strings | Lines merged into target `.gitignore`. |
| `setup` | no | object | Optional script filenames per lifecycle phase. |
| `hints` | no | object | Optional **`global`** / **`local`** lists of plain-text agent hints (see §4.9). |

Unknown top-level keys may be rejected or warned in strict validation in future; stick to the keys above.

### 4.2 `files` — keys and values

- **Key:** path relative to **target repository root**, POSIX style (e.g. `spec/main.md`).
- **Template on disk:** `extsrc/files/<path>` — same relative path (e.g. `extsrc/files/spec/main.md`).

**Value fields:**

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `description` | string | optional | Human label; **required** (non-empty) if `globalRead` or `localRead` is not `no` under strict checks. |
| `mode` | string | `static` | `static` or `artifact` — see §4.4. |
| `globalRead` | string | `no` | `required`, `auto`, or `no` — navigation / global index (§4.5). |
| `localRead` | string | `no` | `required`, `auto`, or `no` — how this file appears in rendered skills for **this** extension (§4.5). |

### 4.3 `folders` — keys and values

- **Key:** path relative to **target repository root** (directory).
- **Value:** object with **`mode`**: **`static`** or **`artifact`** (default `static` if omitted in implementations that default it).

Use **`folders`** to declare directories the methodology owns for **removal/update policy** (static extension-owned trees vs artifact project-owned trees). Concrete creation of empty dirs may depend on tooling; declare paths that your methodology and scripts rely on.

### 4.4 `mode`: `static` vs `artifact` (**files**)

- **`static`** (default): template under `extsrc/files/` is copied to the target on install. On later updates, the tool may **overwrite** the target file with the template (typically with a warning if the file already existed).
- **`artifact`**: template is copied **only if** the target file **does not exist**. After that, the file is treated as **project-owned**; extension updates must **not** overwrite it. Use for tasks, plans, repo-specific docs.

### 4.5 `globalRead` and `localRead`

Allowed values: **`required`**, **`auto`**, **`no`** (default **`no`**).

**`globalRead`** — whether the path is listed in **`spawn/navigation.yaml`** for the whole repo:

- `required` — treat as must-read for every session context.
- `auto` — listed as optional contextual reading.
- `no` — omit from global navigation index for this declaration.

**`localRead`** — whether/how the same path is injected into **rendered skills** belonging to **this** extension:

- `required` — mandatory reads when using those skills.
- `auto` — contextual reads block.
- `no` — omit from local skill read lists from this file entry.

**Skill-only `required-read`:** Under **`skills`**, the **`required-read`** list adds mandatory paths for **that skill** only. Spawn merges global required, local required, and skill `required-read`, then **deduplicates**. Avoid duplicating entries already covered by `globalRead: required` or `localRead: required` on `files` — redundant but harmless.

### 4.6 `skills` — keys and values

- **Key:** exact **filename** in `extsrc/skills/`, including **`.md`** (e.g. `run-task.md`).
- **Value fields:**

| Field | Type | Purpose |
|-------|------|---------|
| `name` | string (optional) | Skill display id; overrides frontmatter and filename heuristic. |
| `description` | string (optional) | Short summary; overrides frontmatter. |
| `required-read` | list of strings | Target-relative paths mandatory **for this skill** in addition to merged global/local rules. |

### 4.7 `agent-ignore` and `git-ignore`

- **`agent-ignore`:** glob strings merged into the IDE’s ignore mechanism so agents skip large or irrelevant trees.
- **`git-ignore`:** lines appended to the target `.gitignore`.

### 4.8 `setup` — lifecycle scripts

Optional object. Each value is a **filename** under **`extsrc/setup/`**. If the key is present, that script **must** exist when checks run in strict mode.

| Key (YAML) | Phase |
|------------|--------|
| `before-install` | Before copying extension into target. |
| `after-install` | After install materialization. |
| `before-uninstall` | Before removing extension (failures may be **blocking** when this hook is defined). |
| `after-uninstall` | After removal. |
| `healthcheck` | Post-install verification (`spawn extension healthcheck <name>` from target). |

Scripts are **trusted code**. They should be **idempotent**, avoid silently deleting user **artifact** data, and fail loudly when migration is unsafe.

### 4.9 `hints` — optional agent reminders

Optional top-level map. Values are **lists of plain strings** (YAML keys **`global`** / **`local`** → `hints.global` / `hints.local`). Content is **plain text** only (no indirection into other files).

| | **`global`** | **`local`** |
|---|--------------|-------------|
| **`spawn/navigation.yaml`** | Mirrored on this pack’s **`read-required`** **`- ext:`** block as **`hints`** | Not written |
| **Rendered skills** | Every skill includes **all** packs’ **`hints.global`**, in installed-extension order (same breadth idea as flattened global mandatory reads), then this pack’s **`local`**, then maintainer **`read-required` → `rules` → `hint`** | Only skills **owned by this pack** |
| **AGENTS / entry rollup** | Included (all packs + maintainer hints) | Omitted |

Strip and dedupe by exact string (first occurrence wins). Oversize strings may emit **`SpawnWarning`**; persisted navigation may truncate past **512** Unicode codepoints. Rendered skill **Hints** blocks may truncate overall size; **`AGENTS.md`** keeps full hint text but can still warn on the same measurement thresholds.

## 5. Skill sources (`extsrc/skills/*.md`)

### 5.1 Format

- Encoding: **UTF-8**.
- Format: **Markdown**.
- **Filename** (e.g. `my-task.md`) **must** match a key under **`skills:`** in `config.yaml` (if you declare skills in config). Skills are discovered from installed `extsrc/skills/*.md` together with config.

### 5.2 Optional YAML frontmatter

If the file begins with a line that is exactly **`---`**, the next line **`---`** that appears alone on a line closes the frontmatter block. The block between them is parsed as **YAML** (safe subset).

Common frontmatter keys:

| Key | Purpose |
|-----|---------|
| `name` | Skill name if not set in `config.yaml`. |
| `description` | Short description if not set in `config.yaml`. |

If there is no valid frontmatter block, the **entire file** is the skill **body**.

### 5.3 Resolving `name` and `description`

**Name:**

1. `config.yaml` → `skills` → this file’s entry → **`name`**, if present.
2. Else frontmatter **`name`**, if present.
3. Else the basename of the file **without** `.md`.

**Description:**

1. `config.yaml` → **`description`**, if present.
2. Else frontmatter **`description`**, if present.
3. Else empty string.

### 5.4 Skill body and IDE rendering

The **body** is everything after the closing frontmatter `---`, or the whole file if there is no frontmatter.

Spawn does **not** copy this file as-is into the IDE. It builds **normalized metadata** (name, description, body text, merged required reads, contextual reads) and each **IDE adapter** writes its own format (e.g. per-IDE skill files).

Spawn’s shared **`render_skill_md`** output order is: YAML frontmatter (`name`, `description`), the **skill body**, an optional **Hints** bullet list (cross-extension **`hints.global`**, this pack’s **`hints.local`**, then maintainer **`read-required` rule** hints), **Mandatory reads** (paths + descriptions, with **`spawn/navigation.yaml` last among mandatory bullets), then **Contextual reads** when present. Agents should treat mandatory reads (including the navigation file) as the index for repo-wide context.

Keep the source body **short and procedural**; long reference material should live under `extsrc/files/` and be exposed via `globalRead` / `localRead` or skill `required-read`.

### 5.5 Example skill file

```markdown
---
name: acme-run-task
description: Execute the Acme task flow in the current repo.
---

When the user wants to run an Acme task:

1. Read the mandatory context injected into this skill.
2. Open or create work under `spec/tasks/` as described in the methodology guide.
3. Report blockers explicitly instead of guessing.
```

If `config.yaml` already sets `name` and `description` for this filename, the frontmatter can be omitted.

## 6. Extension MCP (`extsrc/mcp/*.json`)

Use **three** JSON files under **`extsrc/mcp/`**: **`windows.json`**, **`linux.json`**, **`macos.json`**. Each must be valid JSON with the shape below. Spawn selects the file for the host OS when merging MCP. The set of server **`name`** values must be **identical** in all three files (order may differ); **`transport`** (and similar) may differ per platform.

**Do not** add **`extsrc/mcp.json`**: it is obsolete and not read. `spawn extension check` reports it. **`spawn extension init`** and **`spawn extension from-rules`** scaffold **`extsrc/mcp/`** with three empty `{"servers": []}` files.

### 6.1 Not the same as IDE project MCP

Many IDEs use a root **`mcpServers`** object (e.g. Cursor, Claude Code). The **extension** format uses a top-level **`servers` array** in each platform file. Spawn parses the host file, normalizes it, then **adapters** rewrite into each IDE's project format.

### 6.2 Top level (in each platform file)

| Field | Type | Meaning |
|-------|------|---------|
| `servers` | array | Each element is one MCP server. Missing or `[]` is allowed only when all three platform files share the same server **names** (typically all empty). |

### 6.3 Each server object

| Field | Type | Required | Meaning |
|-------|------|----------|---------|
| `name` | string | **yes** | Stable server id. Must be **unique across all installed extensions** in the same target. |
| `transport` | object | implicitly | Connection details; see §6.4. If omitted, defaults are applied inside the parser (`type` defaults to `stdio`). |
| `env` | object | no | Environment variables for the server; see §6.5. |
| `capabilities` | object | no | Feature flags; see §6.6. |

### 6.4 `transport` object

Fields read by the normalizer:

| Field | Default | Meaning |
|-------|---------|---------|
| `type` | `"stdio"` | Transport kind: e.g. **`stdio`**, or remote kinds such as **`sse`**, **`streamable-http`** for URL-based servers (exact strings depend on IDE adapter mapping). |
| `command` | — | Executable for stdio-style servers. |
| `args` | `[]` | Argument list (strings). |
| `cwd` | `"."` | Working directory. |
| `url` | — | URL for HTTP/SSE-style transports when `type` is not plain stdio. |

### 6.5 `env` object

Each **key** is an environment variable name.

Each **value** is either:

**A. A non-object scalar** (string, number, etc.) or any JSON value that is **not** a mapping  

- Treated as a **user-supplied secret**: when writing IDE config, the value is represented as a **placeholder** (e.g. `${VAR}`), not a literal secret. **Never commit real secrets** in `extsrc/mcp/*.json`.

**B. An object** with the following optional fields (defaults shown):

| Field | Type | Default | Meaning |
|-------|------|---------|---------|
| `source` | string | `"user"` | Provenance hint for tooling. |
| `required` | bool | `true` | Whether the variable must be set for the server to work. |
| `secret` | bool | `false` | Whether to mask/placeholder in generated config. |
| `value` | string or null | null | Non-secret default for generated config when `secret` is false. |

### 6.6 `capabilities` object

Optional. Defaults if omitted or partial:

| Field | Default | Meaning |
|-------|---------|---------|
| `tools` | `true` | Server exposes tools. |
| `resources` | `false` | Resources support. |
| `prompts` | `false` | Prompts support. |

### 6.7 Examples

All snippets are the contents of **one** platform file (`windows.json`, `linux.json`, or `macos.json`): top-level **`servers`** array, each server with a **globally unique** `name` in the target (across every installed extension). For a real extension, replicate the same **`servers`** list into **all three** files when transport is identical, or adjust **`transport`** per file when it differs by OS.

**1 — Stdio via `npx`, structured `env` (secret object), explicit `capabilities`:**

```json
{
  "servers": [
    {
      "name": "acme-docs-mcp",
      "transport": {
        "type": "stdio",
        "command": "npx",
        "args": ["-y", "@acme/mcp-server", "--stdio"],
        "cwd": "."
      },
      "env": {
        "ACME_API_TOKEN": {
          "source": "user",
          "required": true,
          "secret": true
        }
      },
      "capabilities": {
        "tools": true,
        "resources": false,
        "prompts": false
      }
    }
  ]
}
```

**2 — Remote HTTP transport (`streamable-http`; confirm the IDE adapter supports this `type`):**

```json
{
  "servers": [
    {
      "name": "acme-remote-mcp",
      "transport": {
        "type": "streamable-http",
        "url": "https://mcp.example.com/v1"
      },
      "capabilities": {
        "tools": true,
        "resources": false,
        "prompts": false
      }
    }
  ]
}
```

**3 — Remote SSE-style URL (`sse` is an alternative remote kind in many stacks; confirm IDE support):**

```json
{
  "servers": [
    {
      "name": "acme-sse-gateway",
      "transport": {
        "type": "sse",
        "url": "https://mcp.example.com/sse"
      }
    }
  ]
}
```

**4 — Stdio via a Python module (interpreter must be on `PATH` where the IDE launches MCP):**

```json
{
  "servers": [
    {
      "name": "acme-python-mcp",
      "transport": {
        "type": "stdio",
        "command": "python",
        "args": ["-m", "acme_mcp", "--stdio"],
        "cwd": "."
      }
    }
  ]
}
```

You can swap `"command": "python"` for e.g. **`uv`** / **`pipx`**-backed wrappers if your team standardizes on them:

```json
"command": "uv",
"args": ["tool", "run", "acme-mcp-cli", "--stdio"]
```

**5 — Minimal stdio (no `env`, no `capabilities` block — defaults apply: tools on):**

```json
{
  "servers": [
    {
      "name": "pack-minimal-stdio",
      "transport": {
        "type": "stdio",
        "command": "npx",
        "args": ["-y", "@some-scope/some-mcp-package"]
      }
    }
  ]
}
```

**6 — Two servers in one platform file (each `name` must still be unique in the whole target); repeat the same structure in the other two platform files:**

```json
{
  "servers": [
    {
      "name": "methodology-docs-mcp",
      "transport": {
        "type": "stdio",
        "command": "npx",
        "args": ["-y", "@org/mcp-docs", "--stdio"]
      }
    },
    {
      "name": "methodology-lint-mcp",
      "transport": {
        "type": "stdio",
        "command": "npx",
        "args": ["-y", "@org/mcp-lint", "--stdio"]
      }
    }
  ]
}
```

**7 — Env shorthand:** use a **plain string** (or any non-object JSON value) as the value for a variable — it is treated like a user-supplied secret placeholder in generated IDE config. **Do not put real tokens in the repo.**

```json
{
  "servers": [
    {
      "name": "pack-with-inline-env-hint",
      "transport": {
        "type": "stdio",
        "command": "npx",
        "args": ["-y", "@org/mcp-server"]
      },
      "env": {
        "ORG_API_KEY": ""
      }
    }
  ]
}
```

An empty string still marks the key for placeholder-style injection in many adapters; prefer the explicit object form from example 1 when you want `required` / `secret` spelled out.

### 6.8 Validation note

`spawn extension check` validates **`extsrc/mcp/`**: all three files present, parseable JSON, matching server **`name`** sets, and no stray **`extsrc/mcp.json`**. Full semantic checks (e.g. missing `name` on a server object) also apply when MCP is loaded for rendering.

## 7. Annotated `config.yaml` example

```yaml
# Current config schema version.
schema: 1

# Installed under spawn/.extend/acme-methodology/
name: acme-methodology

# Compared as a string on extension update.
version: "1.0.0"

hints:
  global:
    - Prefer spectask steps in order.
  local:
    - This skill expects prior spec approval.

# Paths are relative to TARGET repo root.
files:
  methodology/guide.md:
    description: "Core methodology; read when using Acme skills."
    mode: static
    globalRead: auto
    localRead: required
  spec/design/notes.md:
    description: "Team architecture notes (per-repo)."
    mode: artifact
    globalRead: no
    localRead: auto

folders:
  spec/tasks:
    mode: artifact

# Keys must match files in extsrc/skills/
skills:
  acme-run-task.md:
    name: acme-run-task
    description: "Execute the Acme task flow in the current repo."
    required-read:
      - methodology/guide.md

agent-ignore:
  - .acme/cache/**
  - spec/tasks/**/.drafts/**
  - methodology/.local/**

git-ignore:
  - .acme/cache/**
  - spec/tasks/**/.drafts/**

setup:
  after-install: after_install.py
  healthcheck: healthcheck.py
```

**Matching `extsrc/` tree:**

- `extsrc/files/methodology/guide.md`
- `extsrc/files/spec/design/notes.md`
- `extsrc/skills/acme-run-task.md`
- Optional: `extsrc/setup/after_install.py`, `extsrc/setup/healthcheck.py`
- Optional: `extsrc/mcp/*.json` (§6)

## 8. Uniqueness and naming

- After **normalization**, **skill names** must be **unique across all extensions** installed in one target.
- **`servers[].name`** values must be **unique across all extensions** in one target.
- Prefer prefixed names (`acme-run-task`, `acme-docs-mcp`) to avoid collisions when teams combine packs.
- Choose **`files`** / **`folders`** paths that will not clash with other extensions (namespace prefixes).

---

## 9. What the target repository gets

After `spawn extension add ...` (from the target root, after `spawn init`):

1. **Full copy of `extsrc/`** → `<target>/spawn/.extend/<name>/`.
2. **Materialized files**: templates from `extsrc/files/` copied to paths declared in `files`, per `static` / `artifact` rules.
3. **Spawn metadata and IDE outputs** (per installed IDE), e.g.:
   - `spawn/navigation.yaml` (global read index),
   - rendered skills under IDE-specific paths,
   - merged MCP config under IDE-specific paths,
   - agent-ignore and entry-point snippets as implemented by each adapter.

Exact IDE paths and formats are adapter-specific; they are always **derived** from your `config.yaml`, skill sources, and **`extsrc/mcp/*.json`** (host OS file).

## 10. CLI commands (reference)

Run **`spawn`** from the **root of the target repository** (the CLI uses the current working directory; there is typically no `--target` flag).

**Target must be initialized** before extension commands (except commands that only touch an extension source tree):

- **`spawn init`** — create `spawn/` in the current repo.

**Extension source (authoring):**

- **`spawn extension init . --name <id>`** — create `extsrc/` skeleton with empty `config.yaml` and empty **`extsrc/mcp/*.json`** files.
- **`spawn extension check .`** — validate layout; add **`--strict`** to promote issues to errors (missing listed skills, undeclared files under `extsrc/files/`, missing descriptions when read flags are set, invalid or mismatched **`extsrc/mcp/*.json`**, missing setup scripts when referenced).
- **`spawn extension from-rules <source> --name <id> --output <dir>`** — bootstrap `extsrc/` from an existing repo’s `spawn/rules/` (optional workflow).

**Target install:**

- **`spawn extension add <path-or-url> [--branch BR]`** — install/refresh from local path, git URL, or zip URL.
- **`spawn extension list`**, **`spawn extension update <name>`**, **`spawn extension remove <name>`**
- **`spawn extension healthcheck <name>`** — run configured healthcheck script when present.

**Bundled install:**

- **`spawn build install <path-or-url> [--branch BR]`** — install every extension listed in **`extensions.yaml`** at that source root.

## 11. `extensions.yaml` (optional bundle manifest)

A **build** repo lists multiple extension sources so one command installs them all.

Typical shape:

```yaml
extensions:
  - path: https://example.com/org/ext-a.git
    branch: main
  - path: https://example.com/org/ext-b.git
  - path: /absolute/or/relative/path/to/ext-src-or-repo
```

Each entry is resolved to something containing **`extsrc/`** with `config.yaml`. The exact resolution rules follow the same source types as **`spawn extension add`**.

## 12. Typical authoring checklist

1. `spawn extension init . --name my-pack`
2. Add templates under `extsrc/files/`; declare every path in `files` with correct `mode` and read flags.
3. Add `extsrc/skills/*.md`; list each filename under `skills` if you need overrides or `required-read`.
4. Optionally edit **`extsrc/mcp/`** platform files using **`servers`** (§6).
5. Optionally add `setup/` scripts and reference them in `setup:`.
6. `spawn extension check . --strict`
7. In a disposable target: `spawn init`, `spawn extension add <your-source>`, add IDE if needed, verify navigation and skills.
8. Distribute your repo or zip so consumers can pass it to `spawn extension add` or list it in `extensions.yaml`.
